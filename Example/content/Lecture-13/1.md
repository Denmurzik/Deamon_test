# Реализация динамического выделения памяти

Вы уже хорошо знакомы с `malloc`/`free` --- функции выделения и освобождения блоков памяти в куче.

Давайте попробуем разобраться, как сделать свои аналоги этих функций.

Почему может потребоваться реализовывать своё выделение памяти:

1. Нет кучи вообще.
      Такое бывает в программировании для встроенных систем ("*embedded programming*").
      Хотя в этой области предпочитают обходиться без кучи вообще.
2. Куча есть, но работает недостаточно быстро/хорошо.


В этой лекции будем предполагать второй вариант.
В таком случае куча уже есть, но хочется уменьшить количество `malloc`/`free` для улучшения эффективности.

Обычно писать собственное выделение памяти имеет смысл только в том случае,
если нам известно что-то о характере запросов, что неизвестно реализации `malloc`/`free` из `CRT`.
Мы будем далее предполагать, что все выделяемые блоки памяти имеют одинаковый размер.
(хотя многие рассматриваемые приёмы работают и для блоков разного размера)

## Объекты фиксированного размера

Бывает такое, что нужно выделять память под большое количество объектов одного типа.
При этом размеры всех блоков памяти одинаковы, что позволяет выделять память быстрее.

Пусть размер объекта $S$ байт.
Выделим сразу буфер памяти под $B$ блоков --- размера $B \cdot S$ байт:

Будем при каждом выделении памяти возвращать последовательно ячейки массива.
Тогда выделение памяти становится почти бесплатным:
```
my_obj_t* alloc_my_obj() {
    return ptr[k++];
}
```

Конечно, подобный способ выдачи памяти подряд из заранее выделенного куска работает и для объектов разного размера.
Этот способ иногда называют "*linear allocator*", т.к. он линейно идёт по памяти и выдаёт её по мере необходимости.

### Что делать, когда свободные места в буфере закончатся?

Осущестлять `realloc` этого буфера нельзя делать, т.к. адреса выделенных блоков должны сохраняться.

Можно выделить в куче новый буфер, и продолжать работать с ним.
Через какое-то время получится цепочка буферов ("*chunk*"'ов).

Хранение цепочки в связном списке любого вида (лучше, конечно, двусвязный кольцевой с *sentinel*).
В последнем буфере храним количество занятых мест (оно постепенно растёт).

### Что делать с удалением/освобождением объектов?

1. Ничего не делать!
Можно удалить все объекты разом в конце работы: назовём эту операцию `free_all`.
Плюс: это работает очень быстро (кстати, `free` обычно работает медленнее `malloc`).
Минус: в условии постоянного создания и удаления объектов может быть больший перерасход памяти,
т.к. такой аллокатор пустые места не переиспользует.
Требуется памяти на $C$ объектов, где $C$ --- общее количество создаваемых объектов.

Очень удобно для временной памяти, для которой точно известно, когда она станет не нужна, и что её мало.
(кстати, в принципе работает и для блоков разного размера)

2. Можно разрешить "откатывать назад" состояние.
Это работает примерно так:
```
// Сохраняем в mark состояние:
// указатель на текущий буфер
// и количество занятых мест в нём.
mark = save();

/* выделяем временные объекты */

// Возвращаем обратно сохранённое состояние.
rollback(mark);
```

По сути, после отката к какому-то моменту времени все объекты, созданные после него, удаляются автоматически.
Иными словами, разрешено удалять только последний живой объект --- ограничение на порядок удалений.
Сохраняемое сосотяние --- это всего лишь указатель на последний буфер и количество выделенных объектов в нём.

По этому же принципу работает выделение памяти на стеке (для локальных переменных).
Поэтому такой принцип называется "*stack allocator*".
Хорошо сочетается с рекурсией. Работает для аллокации блоков разного размера.

3. Можно отмечать положения удаляемых объектов и переиспользовать их в будущем.
Классический способ реализации называется "*free list*".
Идея такая: будем хранить связный список всех доступных для переиспользования ячеек.
Для экономии памяти, будем хранить каждый узел списка прямо в соответствующей ячейке (ведь они в этот момент не заняты).

```
void alloc() {
    if (!free_list) {
        // Все ячейки нового буфера добавляются в free_list.
        allocate_new_chunk();
    }
    assert(free_list);
    void* res = free_list;
    free_list = free_list->next;
    return res;
}

void free(void* ptr) {
    ((free_node_t*)ptr)->next = free_list;
    free_list = (free_node_t*)ptr;
}
```

Операции по-прежнему работают за $O(1)$, а памяти нужно на $L + B$ объектов,
где $L$ --- максимальное количество живых объектов на какой-либо момент, а $B$ --- размер блока.

Заметим, что ячейки в `free_list` могут попадать в любом порядке.

## Необычное: система слотов.
Имея указатель на потенциально удалённый объект, можно ли определить, жив он или нет?

В случае использования `malloc`/`free` ответ --- нельзя.
Потому что если объект удалён, на его месте могут быть любые данные,
в том числе даже новый объект этого же типа (подозрительной похожий на старый).
Глядя на записанные данные нельзя никак определить подобную ситуаций.
Проблему можно решить, если в ссылках на объект хранить не только адрес/указатель, но ещё дополнительную информацию.

Пример: создание/удаление игровых объектов в `Doom 3`.
Все они хранятся в одном глобальном массиве `entities` размером $4096$ ячеек.
То есть никакой цепочки блоков нет: если все ячейки заняты,
то новые игровые объекты просто перестают создаваться, что является нормальной практикой в мире *gamedev*.

Для каждого объекта будем поддерживать два числа:
* `pos` --- номер ячейки массива `entities`, где лежит объект;
* `spawn_index` --- номер операции создания объекта, которая породила этот объект.

То есть мы будем нумеровать все операции создания игровых объектов по очереди.
Первый созданный объект получает `spawn_index = 0`, второй `spawn_index = 1`, третий `spawn_index = 2`, и так далее.
`spawn_index` показывает момент времени, когда объект был создан
(по аналогии с тем, как работает "время выхода" в обходе графа в глубину).

Будем хранить `spawn_index` и в самом объекте, и в каждой ссылке на него.
В каждой ссылке на игровой объект будем хранить и номер ячейки, и его `spawn_index`:
```
// Ссылка на игровой объект.
struct entity_ptr_t {
    // Номер ячейки в массиве entities, в котором лежал объект
    // (в момент создания ссылки).
    uint16_t pos;

    // spawn_index для того объекта,
    // на который указывает эта ссылка.
    uint16_t spawn_index;
};
```

Тогда если объект был удалён и его место переиспользовано, то в `entities[ptr.pos]` будет записан другой `spawn_index`.
```
entity_t entities[4096];

// "Разыменование" нашей ссылки.
entity_t* get(entity_ptr_t ptr) {
    return &entities[ptr.pos];
}

// Верно ли, что объект по ссылке умер?
int is_ptr_dead(entity_ptr_t ptr) {
    return get(ptr)->spawn_index != ptr.spawn_index;
}

entity_ptr_t create_entity() {
    entity_ptr_t ptr;

    // Свободные ячейки массива хранятся в списке.
    ptr.pos = free_list.pop();

    get(ptr)->spawn_index = ptr.spawn_index = spawn_count++;
    return ptr;
}

void delete_entity(entity_ptr_t ptr) {
    // При удалении очищаем spawn_index.
    get(ptr)->spawn_index = -1;

    free_list.push(ptr.pos);
}
```

## Аллокатор общего назначения

Если есть кусок памяти, и в нём нужно выделять блоки памяти разного размера, то в общем случае ситуация выглядит так:

Весь управляемый отрезок памяти представляет собой последовательность свободных и занятых блоков.
Обычно свободные блоки разделены занятыми --- при освобождении памяти соседние свободные блоки нужно склеивать ("*coalescing*").
Нужно немало дополнительных данных, чтобы работать с этой структурой:
* размеры блоков / связный список всех блоков;
* связный список свободных блоков.

Главная проблема возникает при выделении: нужно найти свободный блок подходящего размера.
Проход по связным спискам может работать за $O(F)$ времени в худшем случае, где $F$ --- количество свободных блоков.
Есть разные стратегии для сокращения этого времени в среднем на практике.

### Что добавляет многопоточность?
Системная куча обычно одна на всю программу, а значит все потоки должны работать с ней одновременно.
При каждом выделении/удалении памяти надо делать блокировку, чтобы избежать состояния гонки при параллельном доступе к куче из нескольких потоков.
Это приводит к дополнительным затратам времени.

## Фрагментация кучи.

Фрагментацией называют ситуацию, когда есть много свободного места, но большой блок памяти выделить не получается.

У 32-битной программы виртуальное адресное пространство 2 GB.
Если есть 10000 блоков памяти, равномерно разбросанных по нему, то массив больше 200 KB выделить уже нельзя.

Возможные последствия фрагментации:
1. Не получается выделить большие блоки памяти (в 64-битной программе может быть не столь актуально).
2. Куча медленно работает, потому что много отдельных свободных блоков.

Что делать?
1. Использовать алгоритмы, которые на практике обычно дают меньшую фрагментацию.
2. Периодически выполнять дефрагментацию кучи.
      Для этого нужно перемещать занятые блоки памяти, что в куче `C` делать никак нельзя
      (зато это делается в `Java`, `C#` и т.п.)

При желании систему умных ссылок и дефрагментации памяти можно реализовать в `C`/`C++`, если очень надо.
Например, такая система используется в играх `Naughty Dog` (говорят, на игровых консолях плохо с фрагментацией).
