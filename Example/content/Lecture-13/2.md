# Управление памятью

Давайте разберём некоторые виды ошибок при работе с памятью в `C`:

    1. Разыменование нулевого указателя:   "`*(char*)0`".
          * Хорошо: почти наверное программа падает.
    2. Обращение вне допустимого диапазона:      к примеру, "`arr[-17]`".
          * Плохо: можно читать чепуху, ломать совсем другие данные, получаются редкие плохо воспроизводимые баги.
          * Хорошо: `Visual C` в конфигурации *Debug* часто выдаёт *heap/stack corruption*.
    3. Чтение неинициализированной памяти:    `malloc` + чтение (или чтение локальных переменных, которым не присвоили значение).
          * Плохо: в релизе будут лежать какие попало данные (обычно нули).
          * Хорошо: `Visual C` в *Debug* заполняет такие данные байтами со значением `0xCC` --- легко заметить и быстро падает.
          Обычно устраняется использованием `calloc` или использованием функций-конструкторов для структур.
    4. Утечка памяти:                 `malloс`, нет `free`.
          * Плохо: память незаметно и неограниченно накапливается, никогда не освобождаясь.
                В конечном итоге это приводит к падению с "*out-of-memory*" (OOM) ошибкой.
    5. Повторное освобождение:                 `malloc` + `free` + `free`.
          * Плохо: можно случайно освободить блок по этому адресу, выделенный кому-то другому.
          * Хорошо: чаще всего всё-таки падает.
    6. Использование после освобождения:              `malloc` + `free` + чтение или запись.
          * Плохо: можно перезаписать совсем другие данные, если этот блок переиспользован, или прочитать чушь.

Далее речь пойдёт в основном о проблемах 4, 5 и 6.
Заметим, что проблема *memory leak* есть не только в `C`/`C++`, но и в языках высокого уровня, т.к. есть ресурсы помимо памяти.
Например, файловые дескрипторы (`FILE*`).

Многие ошибки возникают из-за неопределённого смысла указателя в `C`:
```
void func(/* ... */, int* p);
```
Что означает `p`?

Варианты:
* опциональное входное значение типа `int`;
* входной массив целых чисел `int[]` (возможно опциональный, возможно всегда должен быть не `NULL`);
* выходное значение типа `int` --- в него функция записывает выходные данные (может ли оно быть `NULL`?);
* выходной массив целых чисел (возможно опциональный параметр, возможно нет);
* иногда параметр является и входным, и выходным --- функция обновляет данные.

Чтобы уменьшить путаницу, иногда договариваются имена параметров начинать с буквы
'`i`' (входной), '`o`' (выходной) или '`io`' (и входной и выходной).
Порой также добавляют букву '`a`' (массив), либо '`p`' (указатель).
Например:
```
void compute_prefix_sums(
    int* ia_values,      // Входной массив чисел.
    int i_size,          // Длина массива.
    int* oa_prefix_sums, // Массив префиксных сумм,
                         // заполняется функцией.
    int* iop_total_min   // Текущий минимум среди чисел
);                       // (обновляется,
                         // если в заданном массиве
                         // есть число меньше).
```

Более интересный вопрос:
* удаляет ли функция `compute_prefix_sums` то, на что указывает `ia_values`?

Если посмотреть на пример посложнее (принимает массив, возвращает массив):
```
void do_it(int* ia_input, int** opa_output);
```

Возникают вопросы:
1. Удаляется ли буфер `ia_input` внутри функции `do_it`?
2. Если предположить, что функция записывает в `*opa_output` указатель,
      должен ли вызывающий удалять то, на что он указывает, после вызова?

Эти проблемы можно решить с помощью концепции "владения".
Единоличное владение --- когда у буфера есть ровно один "хозяин", который им распоряжается.
Он должен этот буфер или удалить, или передать владение кому-то другому.
Хозяином может быть другой объект для поля, или текущий *stack frame* для локальной переменной.

Пример (динамический массив):
```
struct array_t {
    int n, cap;
    item_t* arr; // Owning.
};
```

Структура `array_t` описывает динамический массив, который единолично владеет своим буфером памяти `arr`.
Это означает, что в момент исчезновения массива нужно освободить память под `arr`.
Например так:
```
// Надо сказать пользователю, что он должен вызвать clear_array
// в конце использования массива, чтобы освободить память.
void clear_array(array_t* x) {
    x->n = x->cap = 0;
    free(x->arr);
    x->arr = 0;
}
```

или если массивы всегда выделяются на куче:
```
// Надо сказать пользователю, чтобы он всегда удалял массив,
// используя эту функцию free_array.
void free_array(array_t* x) {
    free(x->arr);
    free(x);
}
```

Другой пример:
```
// Записать массив p[0..n) в массив *a.
void assign(array_t* a, item_t* p, int n);
```

Вопрос: если вызывающий сам выделял память под `p`, должен ли он его удалить после вызова?

Варианты ответов на этот вопрос:

1. Да, должен: владение буфером `p` не передаётся в массив при вызове `assign`.
  Это означает, что функция `assign` должна скопировать все элементы (от `p[0]` до `p[n - 1]`) в свой буфер.
  Иначе когда вызывающий удалит `p` (а он может, т.к. он хозяин), массив `*a` окажется сломанным.
```
// Copy-assign.
void assign(array_t* a, item_t* p, int n) {
    ensure_capacity(a, n);
    a->n = n;
    memcpy(a->arr, p, n * sizeof(item_t));
}
```

2. Нет, не должен: при вызове `assign` вызывающий передаёт владение буфером `p` в функцию.
  Тогда массив `*a` становится новым хозяином буфера `p`, и удалять его после вызова снаружи нельзя.
  В этом случае массив может "украсть" этот буфер себе, или "переместить" его (*move*).
  Разумеется, в этом случае вызывать `assign` можно только если `p` указывает на начало буфера в динамической памяти!
```
// Move-assign.
void assign(array_t* a, item_t* p, int n) {
    // Здесь владение буфером a->arr заканчивается,
    // и владение никому не передаётся,
    // значит его необходимо удалить.
    free(a->arr);
    
    // Mассив *a принимает владение буфером p.
    a->n = n;
    a->arr = p;
}
```

Конечно, даже в этом случае можно скопировать буфер, но в этом случае функция должна удалить массив `*a` после копирования.

При любом варианте следует написать комментарий перед функцией, чтобы вызывающий знал, как себя ведёт `assign`.
Вариант 1 можно считать вариантом "по умолчанию" --- владение никуда не передаётся.

В варианте 2 надо написать что-то вроде такого:
```
// Array *a receives ownership over array p
// (it must be allocated with malloc beforehand).
```

Ещё примеры:
```
// Caller receives ownership over the returned string
// (and must delete it using free).
char* strdup(char* s);
  
// No ownership passed, caller must NOT free returned pointer
// (it points to somewhere inside string s or is NULL).
char* strchr(char* s, char ch);

// Caller passes ownership of buff into the function
// (and must not delete it).
// Caller receives ownership of returned memory buffer
// (and must delete it).
void* realloc(void* buff, int size)

// Caller receives ownership over the FILE.
FILE* fopen(char* filename, char* mode);

// Caller passes ownership over the file f.
void fclose(FILE* f);
```

Помимо описанного выше единоличного владения, есть также вариант разделённого ("*shared*") владения.
В этом случае у объекта может быть несколько равноправных хозяев.
Когда хозяину объект становится не нужен, и при этом он последний, кто им владеет, тогда он должен его удалить.
Этот механизм реализуется с помощью подсчёта ссылок ("*reference counting*") --- храним для объекта, сколько у него сейчас хозяев.

Смежный вопрос:
```
array_t a;
init_array(&a);
push_array(&a, 17);
...
array_t b = a;
// ???
```

Что можно сказать о ситуации в конце?

Оба массива указывают на один буфер памяти --- это противоречит концепции единоличного владения!
Если оставить как есть, скорее всего случится *double free* (буфер будет освобождаться и при удалении `a`, и при удалении `b`).
Перед этим может произойти *use-after-free* (если массив `a` будет удалён, а с массивом `b` будут работать).
Либо просто пользователь будет удивляться, почему при изменении `a` изменяется также и `b`.

Разрешить ситуацию можно двумя способами:

1. Занулить `a` после присваивания:
```
array_t b = a;
a.ptr = 0;
a.n = b.cap = 0;
```

  Тогда буфером снова будет владеть один `b` --- получается передача владения без копирования
2. Запретить присваивать массивы, и завести функцию копирования содержимого:
```
void copy_array(array_t* dst, array_t* src) {
    ensure_capacity(dst, src->n);
    dst->n = src->n;
    memcpy(dst->arr, src->arr, src->n * sizeof(item_t));
}
```

Тогда пользователь сможет копировать весь массив (включая содержимое).
После копирования будет два массива, каждый со своим личным буфером.

Более сложная ситуация --- дерево файлов и каталогов.
```
struct file_tree_t {
    // Путь к файлу.
    char* filename;
    
    // Открытый FILE, из которого можно читать fread'ом.
    FILE* fd;
    
    // Первый сын (если это директория).
    file_tree_t* first_son;
    
    // Следующий сын (в данной директории).
    file_tree_t* brother;
    
    // Родительская директория.
    file_tree_t* father;
};

// Корневой узел.
file_tree* root;
```

Как определить, какие указатели в `file_tree_t` отражают владение, какие нет?
Например:
* `filename` --- есть владение: строки пользователя копируются посредством вызова функции `strdup`;
* `fd` --- есть владение: при удалении узла `file_tree_t` нужно закрыть этот файл;
* `first_son`, `brother` --- есть владение: так получается, что каждый узел владеет своими сыновьями;
* `father` --- нет владения: узел не владеет своим отцом.

В таком случае получается, что указатель root косвенно владеет всем деревом!
Можно написать рекурсивную процедуру удаления дерева:
```
void free_tree(file_tree_t* node) {
    free(node->filename);
    if (node->fd) {
        fclose(node->fd);
    }
    free_tree(node->first_son);
    free_tree(node->brother);
    free(node);
}
```

При запуске для корня будет удалено всё дерево.

Можно заметить, что память в куче --- не единственный ресурс, который надо освобождать.
Надо закрывать файлы, освобождать аппаратные ресурсы (например буферы видеодрайвера), убирать *callback*'и и пр.
Таким образом, данная проблема актуальна даже для для языков высокого уровня со сборщиком мусора.

Ещё более сложная проблема --- хранение многогранника.
Он состоит из граней, рёбер и вершин:
```
struct face_t {
    /* ... */
    edge_ptr_array_t boundary;
};

struct edge_t {
    /* ... */
    struct vertex_t *start, *end;
};

struct vertex_t {
    /* ... */
    edge_ptr_array_t incident;
};
```

Легко представить тетраэдр с 4 гранями, 6 рёбрами и 4 вершинами.

Как расставить владение?

1. Указатели в $vertex.incident$ не проявляют владение --- это просто ссылки.
2. Остальные указатели должны проявлять владение: грани владеют рёбрами на своей границе, а рёбра --- своими концевыми вершинами.
      Грань просто не может существовать без рёбер и вершин на своём краю!

Однако видно, что единоличное владение здесь быть никак не может: на каждую вершину указывает три рёбра, и не одно.

Ситуацию можно разрешить двумя способами:
1. Добавляем структуру многогранника, который единолично владеем всеми своими элементами:
```
struct polyhedron_t {
    face_ptr_array_t faces;      // Owning.
    edge_ptr_array_t edges;      // Owning.
    vertex_ptr_array_t vertices; // Owning.
};
```

      Все грани, рёбра и вершины нечем не владеют и ссылаются друг на друга.
      Сам многогранник владеет ими всеми --- они как бы "живут внутри него".
      Естественно, в этом случае в любую модифицирующую операцию надо передавать `polyhedron_t`.

1. Используем раздельное (множественное) владение ("*shared ownership*").
      Каждым ребром вместе владеют обе смежные грани.
      Каждой вершиной вместе равносильно владеют все инцидентные её рёбра.
      У каждого элемента несколько "хозяев", и все они равноправны.

      Удалять элемент нужно в тот момент, когда в него пропадёт последним хозяин.
      Это обычно реализуется с помощью подсчёта ссылок ("*reference counting}*").
      Будем в каждом элементе хранить счётчик ссылок --- сколько у него сейчас хозяев:
```
struct edge_t {
    int ref_count;
    vertex_t* start;
    vertex_t* end;
};
```

Это число надо всегда поддерживать актуальным, а если в какой-то момент оно становится нулевым, то объект надо удалять.
