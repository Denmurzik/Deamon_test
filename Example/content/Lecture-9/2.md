# Динамическое программирование

Это метод решения задач, заключающийся в том, что вместо того чтобы сразу решить стоящую перед нами большую задачу,
мы сначала находим способ разбить её на подзадачи,
такие, что, имея их решения, мы можем по этим решениям найти решение исходной большой задачи.
Затем мы решаем эти подзадачи в определённом порядке (переходя от задач меньшего размера к всё большим задачам),
при этом мы запоминаем решения всех подзадач,
чтобы использовать их повторно при нахождении решений подзадач большего размера.

## Задача о числе разбиений

Требуется найти количество всех различных разбиений заданного числа $N$.
Например $17 = 1 + 5 + 3 + 1 + 7$   --- разбиение числа $17$ на слагаемые.
Если разбиения отличаются только порядком слагаемых, то они считаются одинаковыми:
$1 + 5 + 3 + 1 + 7$ и $1 + 1 + 3 + 5 + 7$ --- это одно разбиение.

Соображение: давайте искать только невозрастающие последовательности слагаемых.
Тогда каждое разбиение будет задаваться только одной последовательностью, а мы как раз последовательности и будем считать.

Пример ($N = 5$):
* $5 = 1 + 1 + 1 + 1 + 1$,
* $5 = 2 + 1 + 1 + 1$,
* $5 = 2 + 2 + 1$,
* $5 = 3 + 1 + 1$,
* $5 = 3 + 2$,
* $5 = 4 + 1$,
* $5 = 5$.

Будем строить разбиения, добавляя по одному числу в последовательность:

```
// На какую сумму числа уже выписали, чтобы в общем получилось N.
Sum = 0
// Все последующие числа должны быть не более Last, чтобы было неубывание.
Last = N

while Sum < N
    // Выбираем очередное число (ВАЖНО: здесь верхняя граница точная).
    choose x in [1 : min(Last, N-Sum)]
    add x to sequence
    // Общая сумма увеличивается на новое число x.
    Sum = Sum + x
    // Все последующие числа должны быть не больше данного числа x.
    Last = x
```

На основе этого можно написать рекурсивный перебор
(числа $Sum$ и $Last$ передаём в рекурсию):

```
rec(Sum, Last)
    if Sum == N
        // Sum = N - конец перебора (построили разбиение). Просто прибавляем 1 к глобальному счётчику.
        Answer++
        return
    // Перебираем все допустимые x.
    for x in [1 : min(Last, N-Sum)]
        // Запускаем перебор хвоста последовательности.
        rec(Sum+x, x)
```

Время работы: $O(A)$, где $A$ --- ответ (число разбиений)
Интересный факт:
    $$A \approx \frac{1}{4 \sqrt{3} N} \exp{(\pi \sqrt{\frac{2}{3}N})} \hspace{0.5em}\text{(при больших $N$)}.$$
При $N > 100$ получится очень большое число.
[Источник](https://en.wikipedia.org/wiki/Partition_function_(number_theory)#Approximation_formulas).

Заметим, наш перебор почти "чистый", то есть он не зависит от глобальных переменных
(увеличение $Answer$ --- единственное исключение).
Значит если в ходе перебора запускается $Rec(Sum=17, Last=10)$ два раза,
то оба раза перебор обходит одинаковые поддеревья и прибавляет к $Answer$ одинаковое $dA$.
Было бы неплохо вычислять это $dA$ только один раз для конкретных $Sum$, $Last$,
а потом использовать уже вычисленный результат.

Сделаем перебор полностью чистым: пусть он возвращает $dA$ как return value, а глобальную переменную $Answer$ уберём.
Все результаты будем запоминать в двойном массиве $memo$.

```
memo[0 : N][0 : N]

rec(Sum, Last)
    // -1 в memo означает, что значение ещё не вычислено.
    if memo[Sum][Last] == -1
        if Sum == N
            // Разбиение построено: один вариант.
            res = 1
        else
            $res = 0
            // Перебираем все допустимые x.
            for x in [1 : min(Last, N-Sum)]
                // Запускаем перебор хвоста последовательности.
                res = res + rec(Sum+x, x)
        // Запоминаем результат в memo.
        memo[Sum][Last] = res
    // Возвращаем запомненный результат.
    return memo[Sum][Last]
```

Это называется: перебор "с мемоизацией".

Оценим время работы:
1. Для конкретных значений $Sum$ и $Last$ вычисление происходит максимум один раз.
2. Код вычисления работает за $O(N)$ --- цикл по $x$.

Получается время работы $O(N^3)$ --- намного меньше, чем $\exp{(\sqrt{N})}$.

Теперь построим решение методом "динамического программирования".
Сначала определим, что именно возвращает $Rec(Sum, Last)$:
$R[Sum, Last]$ --- это количество неубывающих последовательностей (хвостов), таких что:
* все числа не превышают $Last$;
* сумма чисел равна $N - Sum$.

Внутри $Rec$ написан код, который вычисляет $R[Sum, Last]$ через $R[*, *]$ для других параметров.
Можно переписать его в виде явной формулы (просто смотрим, как вычисляет res код, или думаем), получится:
* база: $R[Sum, Last] = 1$ при $Sum = N$;
* переход индукции: $R[Sum, Last] = \sum\limits_{x=1}^{min(Last, N-Sum)} R[Sum + x, x]$ при $Sum \neq N$.

Это называется "рекуррентной формулой", т.к. значение $R[Sum, Last]$ зависит только от $R[S, L]$ для $S > Sum$
(т.е. только от "предыдущих значений").
Очевидно, что все значения $R[*, *]$ можно посчитать по этой формуле, если перебирать в порядке уменьшения $Sum$.

Получается такой алгоритм:

```
res[0 : N][0 : N]

for Sum in [N : 0]
    for Last in [0 : N]
        if Sum == N
            res[Sum][Last] = 1
        else
            res[Sum][Last] = 0
            for x in [1 : min(Last, N-Sum)]
                $res[Sum][Last] = res[Sum][Last] + res[Sum+x][x]
```

Это работает за $O(N^3)$.

Подобное решение называется решением методом динамического программирования.
Оно работает очень похоже на перебор с мемоизацией, только:
1. В ДП можно обойти все состояния простыми циклами, в переборе нужна рекурсия.
2. В ДП нужно определить порядок вычисления ответов,
      перебор с мемоизацией определяет его самостоятельно (естественно, циклических зависимостей быть не должно).
3. В ДП вычисляются все ответы, а в переборе с мемоизацией только те, которые реально нужны.

Поэтому перебор с мемоизацией иногда называют "ленивым динамическим программированием".

## Задача о рюкзаке}

Есть $N$ предметов, у каждого вес $w_i$ и стоимость $c_i$.
Есть рюкзак, который выдерживает максимальный вес $W_{max}$ (вместимость рюкзака).
Нужно решить, какие предметы взять, так чтобы:
1. Они входили в рюкзак (суммарный вес не больше вместимости).
2. Их общая стоимость была максимально возможной.

Математически эта задача записывается в таком виде:
* есть $N$ переменных $x_i \in \{0, 1\}$;
* $\sum{x_i w_i} \le W_{max}$;
* $\sum{x_i c_i} \rightarrow max$.

Можно построить любое решение, перебирая предметы по очереди:

```
sumW = 0
sumC = 0
for k in [0 : N-1]
    // Выбираем: взять k-ый предмет или нет.
    choose X_k in {take, skip}
    if X_k = take and sumW + w_k <= W_max
        sumW = sumW + w_k
        sumC = sumC + c_k
    choose solution with max sumC
```

Это можно превратить в рекурсивный перебор.
Попробуем сделать рекурсивную функцию "чистой":
1. Заметим, что в переборе нужна информация о параметрах:
      $k$ --- количество предметов, которые мы просмотрел
      $sumW$ --- суммарный вес выбранных предметов среди них
      $sumC$ --- суммарная стоимость выбранных предметов
2. Допустим, среди оставшихся $N-k$ предметов мы наберём на стоимость $addC$.
      Тогда общая стоимость будет $sumC + addC$, и $sumC$ больше ни на что не влияет.
      Пусть тогда перебор возвращает оптимальное значение $addC$ как return value.

Псевдокод:

```
rec(k, sumW)
    if k == N
        // Предметов не осталось, значит общая стоимость остатка нулевая.
        return 0
    // Пробуем не брать k-ый предмет.
    $C_skip = rec(k+1, sumW)
    // Брать предмет можно, только если он входит в рюкзак.
    if sumW + w_k <= W_max
        // Если предмет возьмём, то надо выбрать оптимальный набор среди оставшихся предметов (начиная с k+1-ого),
        // и не забываем добавить стоимость k-ого предмета.
        C_take = rec(k+1, sumW + w_k) + c_k
    return max(C_skip, C_take)
```

Получается перебор за $O(2^N)$ --- на каждом из $N$ шагов два варианта выбора.

Пусть веса $w_i$, $W_{max}$ целые, и не превышают какого-то числа (например, $10000$).
Тогда среди перебираемых вариантов $sumW$ будет целым и не более $W$, а значит будет часто совпадать.

Добавим мемоизацию в перебор:
1. При первом вызове $Rec(k, sumW)$ сохряняем вычисленное значение в $memo[k][sumW]$.
2. При повторых вызовах этой функции с теми же значениями аргументов вместо вычислений достаём уже рассчитанное значение из $memo$.

Время работы будет $O(N \cdot W)$, т.к. столько элементов в $memo$, и для вычисления каждого нужно $O(1)$ действий.

Как восстанавливать решение (т.е. какой набор предметов надо брать)?
Примерно так же, как это делалось в алгоритмах поиска кратчайшего пути в графе.
Нужно запоминать для каждого состояния $(k, sumW)$, какой выбор в нём оптимальный.

То есть когда мы находим максимум из $C_{skip}$ и $C_{take}$
нужно записать "$F[k, sumW] \gets take$", если выбрали $C_{take}$, или "$F[k, sumW] \gets skip$" иначе.

То есть:
    $F[k, sumW] \in {take, skip}$ --- как был получен оптимальный ответ для $R[k, sumW]$
                                   нужно ли брать $k$-ый предмет для оптимальности решения?

Заметим, что $F[k, sumW]$ следует запоминать в тот же момент, когда мы вычисляем $Rec(k, sumW)$.
Это легко сделать и для перебора с мемоизацией, и для динамического программирования.

Зная $F$ для всех состояний, можно проследить, как менялось состояние по предметам:

$(0, 0) \xrightarrow{F[0,0]=take} (1, w_0) \xrightarrow{F[1,w_0]=skip} (2, w_0) \xrightarrow{F[2,w_0]=take} (3, w_0+w_2) \rightarrow ...$.
